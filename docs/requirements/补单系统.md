# 补单系统 — 需求文档

---

## 一、业务背景

**业务模式：** 把商品信息给第三方（资源方），第三方拍下商品后留下评论和评论图。

**核心流程：**
1. 录入商品信息（店铺ID、商品名称、商品ID、SKU ID、价格）
2. 指定资源方（第三方）
3. 第三方执行补单操作
4. 记录评论内容和评论图片作为结果
5. 状态更新：待补单 → 已补单 / 取消补单

## 四、数据流

### 新增记录
```
① 点击新增按钮 → ② 填写表单（店铺ID、商品名称、商品ID、SKU ID、价格、评论图片、评论文本、状态）
                 → ③ 提交保存 → ④ 列表刷新
```

### 归组管理
```
① 列表勾选多条记录 → ② 点击「归组」按钮
                     ③ 选择方式：
                        方式A：输入资源方名 → 下拉选择已有组 → 直接绑定
                        方式B：输入资源方名 → 不选下拉 → 创建新组并绑定
                     ④ 确认 → 列表刷新
```

### 取消绑定组
```
① 已归组记录 → ② 点击「取消绑定组」按钮 → ③ 确认 → group_id 置空
                                                   → 组内无记录则自动删除组
```

### 标记发送
```
① 组操作 → ② 标记已发送 → 组内所有记录状态更新为「已补单」
```

## 五、数据库表设计

### 表一：`order_supplement`（补单记录）

```sql
CREATE TABLE order_supplement (
    id          BIGINT AUTO_INCREMENT PRIMARY KEY COMMENT '主键',
    shop_id     VARCHAR(200) NOT NULL COMMENT '店铺标识（平台/店铺名/shopId）',
    product_name     VARCHAR(255) NOT NULL COMMENT '商品名称',
    product_id  VARCHAR(100) NOT NULL COMMENT '商品ID',
    sku_id      VARCHAR(100) DEFAULT NULL COMMENT '商品SKU ID',
    price       DECIMAL(10,2) DEFAULT 0.00 COMMENT '价格',
    review_image    TEXT     DEFAULT NULL COMMENT '评论图片，逗号分隔的相对路径',
    review_text     TEXT     DEFAULT NULL COMMENT '评论文本',
    status      VARCHAR(20)  NOT NULL DEFAULT '待补单' COMMENT '补单状态：待补单/已补单/取消补单',
    group_id    BIGINT       DEFAULT NULL COMMENT '所属组ID',
    create_time DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    update_time DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    INDEX idx_shop_id (shop_id),
    INDEX idx_product_id (product_id),
    INDEX idx_status (status),
    INDEX idx_group_id (group_id),
    FULLTEXT INDEX ft_product (product_name, product_id, sku_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='补单记录表';
```

### 表二：`supplement_group`（组）

```sql
CREATE TABLE supplement_group (
    id             BIGINT AUTO_INCREMENT PRIMARY KEY COMMENT '主键',
    date           DATE         DEFAULT NULL COMMENT '下单日期（归组时填写）',
    resource_party VARCHAR(255) DEFAULT NULL COMMENT '资源方（归组时填写）',
    status         VARCHAR(20)  NOT NULL DEFAULT '待发送' COMMENT '组状态：待发送/已发送',
    create_time    DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    update_time    DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间'
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='补单组';
```

### 组逻辑说明

- 多条记录归为一个组（group_id 指向 supplement_group）
- **归组两种方式：**
  1. **绑定到已有组** — 输入资源方名，下拉模糊搜索已有组（显示 `组ID - 资源方名`），选中后直接绑定
  2. **创建新组** — 自由输入资源方名，确认时创建新组并绑定
- 组默认日期为当天，可修改
- 组状态：待发送 → 已发送（批量更新组内记录状态为「已补单」）
- **取消绑定组** — 单条记录可取消组绑定（group_id 置空），组内无记录时自动删除组
- 列表资源方列通过 group_id JOIN 查询组表显示 `组ID - 资源方名`
- 已归组记录勾选框显示 ✕ 不可选

### 图片存储方案

**根目录：** `C:\Users\46201\Pictures\补单`

**目录结构：** 按日期建子目录

```
C:\Users\46201\Pictures\补单\
├── 2026-07-18\
│   ├── tb35238016_商品名称A_01.jpg
│   ├── tb35238016_商品名称A_02.jpg
│   ├── jgj1101112_商品名称B_01.jpg
│   └── ...
├── 2026-07-19\
│   └── ...
└── ...
```

**命名规则：**
```
{店铺ID}_{商品名称}_{序号}.jpg
```

| 部分 | 说明 | 示例 |
|---|---|---|
| `店铺ID` | 店铺标识 | `tb35238016` |
| `商品名称` | 原始商品名（过长时截取） | `连衣裙` |
| `序号` | 01、02、03… 递增 | `01` |

> **多图存储：** 数据库中 `review_image` 字段存逗号分隔的相对路径
> 格式：`2026-07-18/tb35238016_商品名称A_01.jpg,2026-07-18/tb35238016_商品名称A_02.jpg`
> 完整路径由后端拼接：`C:\Users\46201\Pictures\补单\` + 相对路径
>
> **图片展示（前端映射）：** 前端通过后端文件服务接口展示图片
> - 调用现有接口：`GET /api/ai-image/image-file?path={完整路径}`
> - 前端构造 URL：`/api/ai-image/image-file?path=C:\Users\46201\Pictures\补单\2026-07-18\xxx.jpg`
>
> **上传方式（支持3种，与替换图 UploadModal 一致）：**
> 1. **本地上传** — 点选文件上传
> 2. **图片链接** — 粘贴 URL
> 3. **粘贴图片** — Ctrl+V 粘贴剪贴板图片
>
> **上传流程：**
> 1. 弹出 UploadModal → 用户选择/粘贴图片
> 2. UploadModal 读取为 base64 数据
> 3. 前端调用后端上传接口，发送 base64
> 4. 后端自动按日期建子目录，按 `{店铺ID}_{商品名称}_{序号}.jpg` 命名保存
> 5. 后端返回相对路径
> 6. 前端将路径追加到表单的评论图片字段（多张以逗号分隔）
